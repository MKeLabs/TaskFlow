## Concepts Primer — Signals and Observables

> Read this before starting. These two concepts appear in every exercise.

### What is a Signal?

A **signal** is a reactive value container introduced in Angular 16. It holds a value and automatically notifies the template (and any `computed()` or `effect()` that reads it) when the value changes — without you having to call `detectChanges()` or use `async` pipes.

```ts
// Create a signal with an initial value
readonly count = signal(0);

// Read the current value — call it like a function
console.log(this.count()); // 0

// Replace the value
this.count.set(5);

// Derive a new value from the current one
this.count.update(n => n + 1);
```

In templates, you also call it like a function:

```html
<p>{{ count() }}</p>
<button [disabled]="busy()">Save</button>
```

**When Angular sees `count()` in a template, it tracks it.** The next time `count.set(...)` is called, only the parts of the template that read that signal are re-rendered.

**Common patterns you'll see in this codebase:**

| Pattern | Meaning |
|---|---|
| `signal(false)` | A boolean flag (loading, busy, open…) |
| `signal<string \| null>(null)` | An optional value — `null` means "nothing yet" |
| `signal<Item[]>([])` | A list, starts empty |
| `items.update(list => [newItem, ...list])` | Prepend an item without replacing the whole array |
| `computed(() => this.token() ?? '')` | A derived read-only value — recalculates when its dependencies change |

---

### What is an Observable? What about Promises?

Angular's `HttpClient` returns **Observables** (from the RxJS library), not Promises.

A **Promise** resolves once and is done:

```ts
// Promise — resolves once
fetch('/api/data')
  .then(res => res.json())
  .then(data => console.log(data));
```

An **Observable** is a stream — it can emit multiple values over time, be cancelled, and be composed with operators. For a single HTTP call it behaves similarly to a Promise, but the syntax is different:

```ts
// Observable — you subscribe to receive values
this.http.get<ProjectDto[]>('/api/projects').subscribe({
  next: (data) => console.log(data),   // called when data arrives
  error: (err) => console.error(err),  // called on failure
  complete: () => console.log('done'), // called when the stream ends
});
```

**You must call `.subscribe()` to trigger the HTTP request.** An Observable is lazy — it does nothing until subscribed.

In this codebase the pattern you'll see in every component is:

```ts
this.busy.set(true);

this.api.getAll().subscribe({
  next: (items) => this.items.set(items),      // store result in signal
  error: () => this.error.set('Failed.'),      // store error in signal
  complete: () => this.busy.set(false),        // clear loading flag
});
```

The signal updates inside `next` / `error` / `complete` automatically update the template — no extra steps needed.

## Exercise 2 — Handle 401 in the Auth Interceptor

### What is missing?
The current `auth.interceptor.ts` only **attaches** the Bearer token — it does not **react** to 401 responses. If the token expires, the user gets a silent error or an empty list, not a redirect to login.

### Files to edit
- `src/app/core/auth/auth.interceptor.ts`

### Step-by-step

**Step 1 — Import what you need.**

Add to the imports at the top:
```ts
import { catchError, throwError } from 'rxjs';
import { Router } from '@angular/router';
```

**Step 2 — Inject `Router` inside the interceptor function.**

Angular functional interceptors use `inject()` at call time:
```ts
const router = inject(Router);
```

**Step 3 — Detect whether the request is an auth call.**

Auth endpoints (`/api/auth/login`, `/api/auth/register`) returning 401 should NOT trigger a redirect loop:
```ts
const isAuthCall = req.url.startsWith('/api/auth/login') || req.url.startsWith('/api/auth/register');
```

**Step 4 — Pipe `catchError` onto the response.**

After `next(withAuthReq)` (or `next(req)`), add:
```ts
return next(withAuthReq).pipe(
  catchError((err) => {
    if (err.status === 401 && !isAuthCall) {
      auth.logout();
      void router.navigate(['/login'], { queryParams: { returnUrl: router.url } });
    }
    return throwError(() => err);
  })
);
```

> `throwError(() => err)` re-throws so downstream subscribers (like the component's `error` signal) still receive the error.

### Full replacement for `src/app/core/auth/auth.interceptor.ts`

```ts
import { HttpInterceptorFn, HttpRequest } from '@angular/common/http';
import { inject } from '@angular/core';
import { Router } from '@angular/router';
import { catchError, throwError } from 'rxjs';
import { AuthService } from './auth.service';

export const authInterceptor: HttpInterceptorFn = (req, next) => {
  const auth = inject(AuthService);
  const router = inject(Router);
  const token = auth.token();
  const isAuthCall =
    req.url.startsWith('/api/auth/login') ||
    req.url.startsWith('/api/auth/register');

  const outgoingReq =
    shouldSkipAuthorization(req.url) || !token
      ? req
      : withAuthorizationHeader(req, token);

  return next(outgoingReq).pipe(
    catchError((err) => {
      if (err.status === 401 && !isAuthCall) {
        auth.logout();
        void router.navigate(['/login'], { queryParams: { returnUrl: router.url } });
      }
      return throwError(() => err);
    })
  );
};

function shouldSkipAuthorization(url: string): boolean {
  return url.startsWith('/assets');
}

function withAuthorizationHeader(req: HttpRequest<unknown>, token: string): HttpRequest<unknown> {
  return req.clone({
    setHeaders: {
      Authorization: `Bearer ${token}`
    }
  });
}
```

**Step 5 — Test with DevTools.**

Run `npm start`. Log in. In the Network tab, find a `/api/projects` request. In Application → Local Storage, delete `taskflow_access_token`. Refresh the projects page — the interceptor should catch the 401 and redirect you to `/login`.

### Why this matters
HTTP interceptors in Angular are the single place to add cross-cutting request/response logic. Centralizing 401 handling here means no component needs to know about token expiry — they just subscribe and the interceptor handles the session cleanup automatically. The `!isAuthCall` guard prevents an infinite redirect loop if the login endpoint itself returns 401.

### Step-by-step

**Step 1 — Create the API service.**

Create `src/app/core/api/task-items.api.ts`. This service mirrors what `ProjectsApi` does — it wraps `HttpClient` calls and defines the DTO shapes as TypeScript interfaces.

```ts
import { Injectable } from '@angular/core';
import { HttpClient } from '@angular/common/http';

export interface TaskItemDto {
  id: number;
  projectId: number;
  projectName: string;
  name: string;
  description: string | null;
  status: number;
  category: number;
  dueDate: string | null;
  tags: { id: number; name: string }[];
  comments: { id: number; content: string }[];
}

export interface CreateTaskItemRequest {
  projectId: number;
  name: string;
  description: string | null;
  status: number;
  category: number;
  dueDate: string | null;
  tagIds: number[];
}

// Human-readable labels for the enum values the API sends as numbers.
export const TASK_STATUS_LABELS: Record<number, string> = {
  0: 'Todo',
  1: 'In Progress',
  2: 'Done',
  3: 'Blocked',
};

export const TASK_CATEGORY_LABELS: Record<number, string> = {
  0: 'General',
  1: 'Bug',
  2: 'Feature',
  3: 'Chore',
};

@Injectable({ providedIn: 'root' })
export class TaskItemsApi {
  constructor(private readonly http: HttpClient) {}

  getAll() {
    return this.http.get<TaskItemDto[]>('/api/taskitems');
  }

  create(request: CreateTaskItemRequest) {
    return this.http.post<TaskItemDto>('/api/taskitems', request);
  }
}
```

> **Why `Record<number, string>`?** C# enum values serialize as integers by default. Rather than mapping them in every template, we centralize the label lookup in the API file next to the interface that defines the shape.

**Step 2 — Create the component class.**

Create `src/app/pages/tasks/tasks.component.ts`. Use Angular signals for all reactive state — the same pattern used in `ProjectsComponent`.

```ts
import { Component, signal } from '@angular/core';
import { CommonModule } from '@angular/common';
import { FormsModule } from '@angular/forms';
import { RouterLink } from '@angular/router';
import {
  TaskItemsApi,
  TaskItemDto,
  CreateTaskItemRequest,
  TASK_STATUS_LABELS,
  TASK_CATEGORY_LABELS,
} from '../../core/api/task-items.api';
import { ProjectsApi, ProjectDto } from '../../core/api/projects.api';

@Component({
  selector: 'app-tasks',
  standalone: true,
  imports: [CommonModule, FormsModule, RouterLink],
  templateUrl: './tasks.component.html',
  styleUrls: ['./tasks.component.scss'],
})
export class TasksComponent {
  readonly busy = signal(true);
  readonly creating = signal(false);
  readonly error = signal<string | null>(null);
  readonly tasks = signal<TaskItemDto[]>([]);
  readonly projects = signal<ProjectDto[]>([]);

  // Single DTO signal — shape always matches what the API expects.
  readonly newForm = signal<CreateTaskItemRequest>({
    projectId: 0,
    name: '',
    description: null,
    status: 0,
    category: 0,
    dueDate: null,
    tagIds: [],
  });

  readonly statusLabels = TASK_STATUS_LABELS;
  readonly categoryLabels = TASK_CATEGORY_LABELS;
  readonly statusOptions = Object.entries(TASK_STATUS_LABELS).map(([v, l]) => ({ value: +v, label: l }));
  readonly categoryOptions = Object.entries(TASK_CATEGORY_LABELS).map(([v, l]) => ({ value: +v, label: l }));

  constructor(
    private readonly api: TaskItemsApi,
    private readonly projectsApi: ProjectsApi,
  ) {
    this.loadProjects();
    this.load();
  }

  loadProjects() {
    this.projectsApi.getAll().subscribe({
      next: (items) => this.projects.set(items),
    });
  }

  load() {
    this.error.set(null);
    this.busy.set(true);

    this.api.getAll().subscribe({
      next: (items) => this.tasks.set(items),
      error: () => this.error.set('Could not load tasks. Are you logged in?'),
      complete: () => this.busy.set(false),
    });
  }

  // Named handlers keep the template clean — no inline logic in (ngModelChange).
  onProjectIdChange(value: string | number) {
    this.newForm.update((f) => ({ ...f, projectId: Number(value) || 0 }));
  }

  onNameChange(value: string) {
    this.newForm.update((f) => ({ ...f, name: value || '' }));
  }

  onDescriptionChange(value: string) {
    this.newForm.update((f) => ({ ...f, description: value || null }));
  }

  onStatusChange(value: string | number) {
    this.newForm.update((f) => ({ ...f, status: Number(value) || 0 }));
  }

  onCategoryChange(value: string | number) {
    this.newForm.update((f) => ({ ...f, category: Number(value) || 0 }));
  }

  createTask() {
    this.error.set(null);
    const form = this.newForm();

    if (!form.projectId || form.projectId <= 0) {
      this.error.set('Please select a project.');
      return;
    }

    if (!form.name.trim()) {
      this.error.set('Task name is required.');
      return;
    }

    const request: CreateTaskItemRequest = {
      ...form,
      name: form.name.trim(),
      description: form.description?.trim() || null,
    };

    this.creating.set(true);
    this.api.create(request).subscribe({
      next: (created) => {
        this.tasks.update((items) => [created, ...items]);
        // Reset by writing a fresh default — no need to clear each field separately.
        this.newForm.set({ projectId: 0, name: '', description: null, status: 0, category: 0, dueDate: null, tagIds: [] });
      },
      error: () => this.error.set('Could not create task.'),
      complete: () => this.creating.set(false),
    });
  }
}
```

> **Why named handler methods instead of inline `update()` in the template?** Keeping `(ngModelChange)="onNameChange($event)"` instead of `(ngModelChange)="newForm.update(f => ({ ...f, name: $event }))"` makes templates easier to read and makes the component easier to test. The logic lives in one place (the class), not scattered across the HTML.

> **Why reset the form after success instead of reloading?** Calling `load()` again would trigger a full round-trip. Prepending the returned `created` object from the API (`this.tasks.update(items => [created, ...items])`) gives instant feedback with no extra network request.

**Step 2 — Create the detail component class.**

Create `src/app/pages/tasks/task-detail.component.ts`.

The component reads the `:id` route parameter, calls `getById`, and stores the result as a single `signal<TaskItemDto | null>`. Notice there is no separate signal per field — the entire API response lives in one signal.

```ts
import { Component, signal } from '@angular/core';
import { CommonModule } from '@angular/common';
import { ActivatedRoute, RouterLink } from '@angular/router';
import { TaskItemsApi, TaskItemDto, TASK_STATUS_LABELS, TASK_CATEGORY_LABELS } from '../../core/api/task-items.api';

@Component({
  selector: 'app-task-detail',
  standalone: true,
  imports: [CommonModule, RouterLink],
  templateUrl: './task-detail.component.html',
  styleUrls: ['./task-detail.component.scss'],
})
export class TaskDetailComponent {
  readonly busy = signal(true);
  readonly error = signal<string | null>(null);
  readonly task = signal<TaskItemDto | null>(null);

  readonly statusLabels = TASK_STATUS_LABELS;
  readonly categoryLabels = TASK_CATEGORY_LABELS;

  constructor(
    private readonly api: TaskItemsApi,
    private readonly route: ActivatedRoute
  ) {
    const id = Number(this.route.snapshot.paramMap.get('id'));
    this.load(id);
  }

  private load(id: number) {
    this.api.getById(id).subscribe({
      next: (item) => this.task.set(item),
      error: () => this.error.set('Task not found or you do not have access.'),
      complete: () => this.busy.set(false),
    });
  }

  // Helper keeps tag-joining logic out of the template expression.
  getTagNames(task: TaskItemDto): string {
    return task.tags?.length ? task.tags.map((tag) => tag.name).join(', ') : '—';
  }
}
```

> **Why `signal<TaskItemDto | null>`?** Before the API responds, there is no task to show — `null` represents the "not yet loaded" state. The template checks `*ngIf="task() as t"` before rendering any fields. This is cleaner than a separate `loaded` boolean flag.

> **Why `snapshot.paramMap` instead of subscribing to `paramMap`?** For a detail page that is always fully replaced when navigating to a new ID, the snapshot is sufficient and simpler. Observable `paramMap` is only needed if the same component instance reuses across ID changes (e.g. next/previous navigation without destroying the component).

> **Why `getTagNames()` as a helper instead of an inline expression?** `task.tags.map(...).join(', ')` in the template would work, but it creates a new array on every change detection cycle. A named method makes it easy to spot, test, and replace later.

**Step 5 — Update the guard to branch between `/login` and `/unauthorized`.**

In `src/app/core/auth/auth.guard.ts`:

```ts
import { CanActivateFn, Router } from '@angular/router';
import { inject } from '@angular/core';
import { AuthService } from './auth.service';

export const authGuard: CanActivateFn = (_route, state) => {
  const auth = inject(AuthService);
  const router = inject(Router);

  if (auth.isAuthenticated()) {
    return true;
  }

  // Direct visits to /projects show the "unauthorized" page.
  if (state.url.startsWith('/projects')) {
    return router.createUrlTree(['/unauthorized']);
  }

  // Other protected routes redirect to login with a return URL.
  return router.createUrlTree(['/login'], { queryParams: { returnUrl: state.url } });
};
```

**Step 6 — Verify manually.**

Run `npm start`. Without logging in, navigate to `http://localhost:4200/projects`. You should land on `/unauthorized`. Navigate to an unknown path like `/foo` — the wildcard should redirect to `/projects`, which then guards and shows `/unauthorized`. Log in and navigate to `/projects` — it should work normally.

### Why this matters
`CanActivateFn` returns either `true`, `false`, or a `UrlTree`. Returning a `UrlTree` (via `router.createUrlTree(...)`) is the modern Angular way to perform guard-based redirects — it lets the router handle the navigation cleanly instead of side-effecting with `router.navigate()`. The two-branch guard teaches students that "not authenticated" and "not authorized" are conceptually different states that may warrant different UX.
