# Feature 10 — Comment Ownership: Delete Restriction

This document covers the change that prevents users from deleting comments they did not write. Admins can delete any comment; regular users can only delete their own.

---

## The problem

After Feature 9, any authenticated user could call `DELETE /api/taskcomments/{id}` and remove any comment. There was no record of who wrote a comment, so there was nothing to check against.

---

## What we are changing

| Layer | Change |
|---|---|
| Database | New nullable column `CreatedByUserId` on `TaskComments` |
| Entity | `TaskCommentEntity.CreatedByUserId` property |
| BLL DTO | `TaskCommentDto` gains `CreatedByUserId` |
| BLL Service | `CreateAsync` stores the author; new `GetByIdAsync` to fetch a single comment |
| API Controller | `Create` captures userId from JWT; `Delete` checks ownership before proceeding |
| Angular API type | `CommentDto` and `TaskItemDto.comments` gain `createdByUserId` |
| Angular `AuthService` | New `userId` computed signal (reads `sub` from JWT) |
| Angular template | Delete button is hidden for comments the current user did not write |

---

## How the user ID is stored and verified

The user ID comes from the JWT that the client sends on every request. When a user logs in the backend creates a token that includes:

```csharp
// TaskFlow.API/Services/JwtTokenService.cs
new(JwtRegisteredClaimNames.Sub, userId),
new(ClaimTypes.NameIdentifier, userId),
```

In a controller, `User.FindFirstValue(ClaimTypes.NameIdentifier)` reads that value back out. We store it when a comment is created, then compare it at delete time.

## Changes: step by step

### Step 1 — Add `CreatedByUserId` to the entity

**File:** `TaskFlow.DAL/Entities/TaskCommentEntity.cs`

**Before:**
```csharp
public class TaskCommentEntity : BaseEntity
{
    public int TaskItemId { get; set; }
    public TaskItemEntity? TaskItem { get; set; }
    public string Text { get; set; } = string.Empty;
}
```

**After:**
```csharp
public class TaskCommentEntity : BaseEntity
{
    public int TaskItemId { get; set; }
    public TaskItemEntity? TaskItem { get; set; }
    public string Text { get; set; } = string.Empty;
    public string? CreatedByUserId { get; set; }
}
```

The property is `string?` (nullable) for two reasons:
1. Comments created before this migration exist in the database without an author — we cannot retroactively fill that value.
2. A `null` value is treated as "unknown author" and no one (except an Admin) is allowed to delete it. This is the safe default.

No navigation property to `ApplicationUser` is added. A FK constraint would complicate the schema and cause issues if a user were ever deleted. Storing the ID as a plain string is sufficient for the ownership check.

### Step 4 — Update `ITaskCommentService` and `TaskCommentService`

**File:** `TaskFlow.BLL/Services/Interfaces/ITaskCommentService.cs`

**Before:**
```csharp
public interface ITaskCommentService
{
    Task<List<TaskCommentDto>> GetByTaskItemIdAsync(int taskItemId, CancellationToken ct = default);
    Task<TaskCommentDto> CreateAsync(TaskCommentCreateDto dto, CancellationToken ct = default);
    Task<bool> DeleteAsync(int id, CancellationToken ct = default);
}
```

**After:**
```csharp
public interface ITaskCommentService
{
    Task<List<TaskCommentDto>> GetByTaskItemIdAsync(int taskItemId, CancellationToken ct = default);
    Task<TaskCommentDto?> GetByIdAsync(int id, CancellationToken ct = default);                      // ← new
    Task<TaskCommentDto> CreateAsync(TaskCommentCreateDto dto, string? createdByUserId, CancellationToken ct = default); // ← new param
    Task<bool> DeleteAsync(int id, CancellationToken ct = default);
}
```

**Why add `GetByIdAsync` here instead of going directly to the repository in the controller?**

Controllers should not depend on repositories. The controller already depends on `ITaskCommentService`; keeping the fetch there avoids introducing a second dependency just for the ownership check.

**File:** `TaskFlow.BLL/Services/Implementations/TaskCommentService.cs`

```csharp
public async Task<List<TaskCommentDto>> GetByTaskItemIdAsync(int taskItemId, CancellationToken cancellationToken = default)
{
    var task = await taskItemRepository.GetByIdWithDetailsAsync(taskItemId, cancellationToken);
    if (task is null) return [];

    return task.Comments
        .Select(x => new TaskCommentDto(x.Id, x.TaskItemId, x.Text, x.CreatedByUserId))
        .ToList();
}

public async Task<TaskCommentDto?> GetByIdAsync(int id, CancellationToken cancellationToken = default)
{
    var entity = await commentRepository.GetByIdAsync(id, cancellationToken);
    return entity is null ? null : new TaskCommentDto(entity.Id, entity.TaskItemId, entity.Text, entity.CreatedByUserId);
}

public async Task<TaskCommentDto> CreateAsync(TaskCommentCreateDto dto, string? createdByUserId, CancellationToken cancellationToken = default)
{
    var entity = new TaskCommentEntity
    {
        TaskItemId = dto.TaskItemId,
        Text = dto.Text,
        CreatedByUserId = createdByUserId   // ← stored here
    };

    await commentRepository.AddAsync(entity, cancellationToken);
    await commentRepository.SaveChangesAsync(cancellationToken);
    return new TaskCommentDto(entity.Id, entity.TaskItemId, entity.Text, entity.CreatedByUserId);
}

public async Task<bool> DeleteAsync(int id, CancellationToken cancellationToken = default)
{
    var entity = await commentRepository.GetByIdAsync(id, cancellationToken);
    if (entity is null) return false;

    commentRepository.SoftDelete(entity);
    await commentRepository.SaveChangesAsync(cancellationToken);
    return true;
}
```

`DeleteAsync` itself stays unchanged — it does not need to know *why* the delete was permitted. The controller is responsible for that decision.

### Step 6 — Update the controller

**File:** `TaskFlow.API/Controllers/TaskCommentsController.cs`

**Before:**
```csharp
[HttpPost]
public async Task<ActionResult<TaskCommentDto>> Create([FromBody] TaskCommentCreateDto dto, CancellationToken cancellationToken)
{
    var created = await taskCommentService.CreateAsync(dto, cancellationToken);
    return Ok(created);
}

[HttpDelete("{id:int}")]
public async Task<IActionResult> Delete(int id, CancellationToken cancellationToken) =>
    await taskCommentService.DeleteAsync(id, cancellationToken) ? NoContent() : NotFound();
```

**After:**
```csharp
[HttpPost]
public async Task<ActionResult<TaskCommentDto>> Create([FromBody] TaskCommentCreateDto dto, CancellationToken cancellationToken)
{
    var userId = User.FindFirstValue(ClaimTypes.NameIdentifier);
    var created = await taskCommentService.CreateAsync(dto, userId, cancellationToken);
    return Ok(created);
}

[HttpDelete("{id:int}")]
public async Task<IActionResult> Delete(int id, CancellationToken cancellationToken)
{
    var comment = await taskCommentService.GetByIdAsync(id, cancellationToken);
    if (comment is null)
        return NotFound();

    var userId = User.FindFirstValue(ClaimTypes.NameIdentifier);
    var isAdmin = User.IsInRole(AppRoles.Admin);

    if (!isAdmin && comment.CreatedByUserId != userId)
        return Forbid();

    await taskCommentService.DeleteAsync(id, cancellationToken);
    return NoContent();
}
```

**Delete flow:**

| Condition | Response |
|---|---|
| Comment does not exist (or is soft-deleted) | `404 Not Found` |
| Requester is an Admin | `204 No Content` — deleted |
| Requester's ID matches `CreatedByUserId` | `204 No Content` — deleted |
| Requester is neither | `403 Forbidden` |
| Comment exists but `CreatedByUserId` is `null` (pre-migration data) | `403 Forbidden` for non-admins — the safe default |

**Why return `403` rather than `404` for comments you don't own?**

Returning `404` would hide the comment's existence from the requester — a common privacy technique. For this app, the comment text is already visible on the page, so there is no information to hide. `403 Forbidden` is the accurate and honest response.

The `using System.Security.Claims;` and `using TaskFlow.DAL.Auth;` imports were added at the top of the file.

### Step 8 — Add `userId` to `AuthService`

**File:** `src/app/core/auth/auth.service.ts`

The existing `parseRoles` function already knows how to decode the JWT payload. We add a similar function for the `sub` (subject) claim, which ASP.NET Identity sets to the user's ID string.

**Add after `parseRoles`:**
```ts
function parseUserId(token: string | null): string | null {
  if (!token) return null;
  try {
    const payload = JSON.parse(atob(token.split('.')[1]));
    return payload['sub'] ?? null;
  } catch {
    return null;
  }
}
```

**Add to `AuthService`:**
```ts
readonly userId = computed(() => parseUserId(this._token()));
```

Like `isAdmin`, this is a `computed` signal — it updates automatically when the token changes (login / logout).

### Step 10 — Update the template

**File:** `src/app/pages/tasks/task-detail.component.html`

**Before:**
```html
<li *ngFor="let c of t.comments" class="comment-item">
  <span class="comment-text">{{ c.text }}</span>
  <button
    class="btn-sm danger"
    type="button"
    (click)="deleteComment(c.id)"
    [disabled]="deletingCommentId() === c.id">
    {{ deletingCommentId() === c.id ? '…' : 'Delete' }}
  </button>
</li>
```

**After:**
```html
<li *ngFor="let c of t.comments" class="comment-item">
  <span class="comment-text">{{ c.text }}</span>
  <button
    *ngIf="isAdmin() || c.createdByUserId === currentUserId()"
    class="btn-sm danger"
    type="button"
    (click)="deleteComment(c.id)"
    [disabled]="deletingCommentId() === c.id">
    {{ deletingCommentId() === c.id ? '…' : 'Delete' }}
  </button>
</li>
```

The `*ngIf` condition mirrors the backend rule exactly:
- `isAdmin()` — show for admins regardless of authorship
- `c.createdByUserId === currentUserId()` — show for the author

If `c.createdByUserId` is `null` (pre-migration comment) and the user is not an admin, `null === someUserId` is `false`, so the button is correctly hidden.

**Note:** This is a UX improvement only. A malicious user could still construct a DELETE request manually. The `403 Forbidden` from the backend is what actually enforces the rule.

---

## Verify it works

1. **Create a comment as User A.** The Delete button should appear next to that comment.
2. **Log in as User B** (non-admin) and view the same task. The Delete button should be hidden for User A's comment.
3. **Log in as Admin** and view the task. The Delete button should appear for all comments.
4. **Attempt a manual DELETE** with User B's token against User A's comment ID (e.g. via Postman). The API should return `403 Forbidden`.
5. **Attempt the same DELETE with Admin's token.** It should return `204 No Content`.
