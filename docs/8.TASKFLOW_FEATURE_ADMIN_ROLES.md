# Feature — Role-Based Route Protection (Admin-Only Projects)

## What we're doing and why

So far every protected route uses `authGuard`, which only checks that the user is **logged in**. But some resources should be restricted further — for example, only admins should be able to manage projects.

We need a second layer of protection:

| Layer | Where it runs | What it checks |
|---|---|---|
| `authGuard` | Angular router (client) | Is the user logged in? |
| `adminGuard` | Angular router (client) | Does the JWT contain the `Admin` role? |
| `[Authorize(Roles = "Admin")]` | ASP.NET Core (server) | **Same check, server-side** — this is the real gatekeeper |

> **Key principle:** The frontend guards are purely a UX improvement. They prevent unnecessary navigation and API calls, but they cannot be trusted for security. The backend `[Authorize(Roles)]` attribute is what actually enforces access control, because the server controls the response.

---

## How roles end up in the JWT

When a user logs in, the backend creates a JWT that includes role claims:

```csharp
// TaskFlow.API/Services/JwtTokenService.cs
claims.AddRange(rolesList.Select(r => new Claim(ClaimTypes.Role, r)));
```

ASP.NET Core serialises `ClaimTypes.Role` into the JWT under this URI key:
```
http://schemas.microsoft.com/ws/2008/06/identity/claims/role
```

So a decoded JWT payload for an admin looks like:
```json
{
  "sub": "abc123",
  "email": "admin@example.com",
  "http://schemas.microsoft.com/ws/2008/06/identity/claims/role": "Admin"
}
```

And for a regular user the claim is `"User"` or the key is absent entirely.

The `LoginResponse` also returns the roles array directly:
```ts
// auth.models.ts
export interface LoginResponse {
  accessToken: string;
  expiresAtUtc: string;
  email: string;
  roles: string[];  // ← already included
}
```

We can read roles from the token itself — no extra HTTP call needed.

## Changes: step by step

### Step 1 — Decode roles from the JWT in `AuthService`

**Before** (`src/app/core/auth/auth.service.ts`):
```ts
@Injectable({ providedIn: 'root' })
export class AuthService {
  private readonly _token = signal<string | null>(localStorage.getItem(TOKEN_KEY));
  readonly token = computed(() => this._token());
  readonly isAuthenticated = computed(() => !!this._token());
  // ...
}
```

**After:**
```ts
// ASP.NET Identity encodes ClaimTypes.Role under this URI in the JWT payload.
const ROLE_CLAIM = 'http://schemas.microsoft.com/ws/2008/06/identity/claims/role';

function parseRoles(token: string | null): string[] {
  if (!token) return [];
  try {
    const payload = JSON.parse(atob(token.split('.')[1]));
    const raw = payload[ROLE_CLAIM];
    if (!raw) return [];
    return Array.isArray(raw) ? raw : [raw];
  } catch {
    return [];
  }
}

@Injectable({ providedIn: 'root' })
export class AuthService {
  private readonly _token = signal<string | null>(localStorage.getItem(TOKEN_KEY));
  readonly token = computed(() => this._token());
  readonly isAuthenticated = computed(() => !!this._token());
  readonly isAdmin = computed(() => parseRoles(this._token()).includes('Admin')); // ← new
  // ...
}
```

**What `parseRoles` does:**
1. Splits the JWT on `.` — a JWT has three parts: `header.payload.signature`.
2. Takes the middle part (index `[1]`) — that's the base64-encoded payload.
3. Decodes it with `atob()` and parses the JSON.
4. Reads the role claim. The value can be a single string or an array (if the user has multiple roles), so we normalise it to always be an array.
5. Wrapped in `try/catch` — if the token is malformed for any reason, we return an empty array instead of throwing.

`isAdmin` is a `computed` signal, so it automatically updates the moment `_token` changes (login / logout). No subscriptions or manual refresh needed.

### Step 2 — Create the Admin Guard

Create a new file `src/app/core/auth/admin.guard.ts`:

```ts
import { CanActivateFn, Router } from '@angular/router';
import { inject } from '@angular/core';
import { AuthService } from './auth.service';

export const adminGuard: CanActivateFn = (_route, _state) => {
  const auth = inject(AuthService);
  const router = inject(Router);

  if (auth.isAdmin()) {
    return true;
  }

  return router.createUrlTree(['/unauthorized']);
};
```

**Compare with `authGuard`:**

| | `authGuard` | `adminGuard` |
|---|---|---|
| Checks | `auth.isAuthenticated()` | `auth.isAdmin()` |
| On failure redirects to | `/login` (with `returnUrl`) | `/unauthorized` |
| Used on | `/tasks`, `/tasks/:id` | `/projects` |

Why redirect to `/unauthorized` instead of `/login`? Because a non-admin user *is* logged in — sending them to the login page would be confusing. The `/unauthorized` page tells them clearly they don't have the right permissions.

### Step 3 — Protect the backend controller

**Before** (`TaskFlow.API/Controllers/ProjectsController.cs`):
```csharp
[ApiController]
[Route("api/[controller]")]
[Authorize]
public class ProjectsController(IProjectService projectService) : ControllerBase
```

**After:**
```csharp
using TaskFlow.DAL.Auth; // ← add this using

[ApiController]
[Route("api/[controller]")]
[Authorize]                          // any logged-in user can READ
public class ProjectsController(IProjectService projectService) : ControllerBase
{
    [HttpGet]
    public async Task<ActionResult<List<ProjectDto>>> Get(...) => ...

    [HttpGet("{id:int}")]
    public async Task<ActionResult<ProjectDto>> GetById(...) => ...

    [HttpPost]
    [Authorize(Roles = AppRoles.Admin)]   // ← only admins can WRITE
    public async Task<ActionResult<ProjectDto>> Create(...) => ...

    [HttpPut("{id:int}")]
    [Authorize(Roles = AppRoles.Admin)]
    public async Task<IActionResult> Update(...) => ...

    [HttpDelete("{id:int}")]
    [Authorize(Roles = AppRoles.Admin)]
    public async Task<IActionResult> Delete(...) => ...
}
```

**Why split it this way?**

A regular user needs to read the projects list — for example, to populate a dropdown when creating a task. They should never need to create, rename, or delete a project. Splitting the authorization at the action level captures this exactly:

| Action | Who can call it | Why |
|---|---|---|
| `GET /api/projects` | Any authenticated user | Needed to populate dropdowns |
| `GET /api/projects/{id}` | Any authenticated user | Read-only detail |
| `POST /api/projects` | Admin only | Creating projects is privileged |
| `PUT /api/projects/{id}` | Admin only | Renaming/editing is privileged |
| `DELETE /api/projects/{id}` | Admin only | Deletion is privileged |

`AppRoles.Admin` is a constant (`"Admin"`) defined in `TaskFlow.DAL/Auth/AppRoles.cs`. Using the constant instead of a string literal prevents typos and makes refactoring easier.

When a non-admin hits a write endpoint:
- `401 Unauthorized` — no token at all
- `403 Forbidden` — valid token but wrong role

The route guard already blocks navigation, but we should also hide the link — there's no point showing users a link they can't use.

**Before** (`src/app/app.component.ts`):
```ts
export class AppComponent {

}
```

**After:**
```ts

  readonly isAuthenticated = computed(() => this.auth.isAuthenticated());
  readonly isAdmin = computed(() => this.auth.isAdmin()); // ← add
}
```

**Before** (`src/app/app.component.html`):
```html
<nav class="nav">
  <a routerLink="/projects" routerLinkActive="active">Projects</a>
  <a routerLink="/tasks" routerLinkActive="active">Tasks</a>
  <a routerLink="/login" routerLinkActive="active">Login</a>
</nav>
```

**After:**
```html
<nav class="nav">
  @if (isAdmin()) {
    <a routerLink="/projects" routerLinkActive="active">Projects</a>
  }
  <a routerLink="/tasks" routerLinkActive="active">Tasks</a>
  <a routerLink="/login" routerLinkActive="active">Login</a>
</nav>
```

`@if` is Angular's built-in control flow (available since Angular 17). It conditionally renders the element — no `*ngIf` directive needed. Because `isAdmin` is a `computed` signal, the template re-evaluates automatically when the user logs in or out.

## Full final state of changed files

### `src/app/core/auth/auth.service.ts`
```ts
import { Injectable, computed, signal } from '@angular/core';
import { HttpClient } from '@angular/common/http';
import { catchError, of, tap } from 'rxjs';
import { LoginRequest, LoginResponse, MeResponse } from './auth.models';

const TOKEN_KEY = 'taskflow_access_token';

const ROLE_CLAIM = 'http://schemas.microsoft.com/ws/2008/06/identity/claims/role';

function parseRoles(token: string | null): string[] {
  if (!token) return [];
  try {
    const payload = JSON.parse(atob(token.split('.')[1]));
    const raw = payload[ROLE_CLAIM];
    if (!raw) return [];
    return Array.isArray(raw) ? raw : [raw];
  } catch {
    return [];
  }
}

@Injectable({ providedIn: 'root' })
export class AuthService {
  private readonly _token = signal<string | null>(localStorage.getItem(TOKEN_KEY));
  readonly token = computed(() => this._token());
  readonly isAuthenticated = computed(() => !!this._token());
  readonly isAdmin = computed(() => parseRoles(this._token()).includes('Admin'));

  constructor(private readonly http: HttpClient) {}

  login(request: LoginRequest) {
    return this.http.post<LoginResponse>('/api/auth/login', request).pipe(
      tap((res) => this.setToken(res.accessToken))
    );
  }

  register(request: LoginRequest) {
    return this.http.post<LoginResponse>('/api/auth/register', request).pipe(
      tap((res) => this.setToken(res.accessToken))
    );
  }

  logout() {
    this.clearToken();
  }

  me() {
    return this.http.get<MeResponse>('/api/auth/me').pipe(
      catchError(() => of({ id: null, email: null, roles: [] } satisfies MeResponse))
    );
  }

  private setToken(token: string) {
    localStorage.setItem(TOKEN_KEY, token);
    this._token.set(token);
  }

  private clearToken() {
    localStorage.removeItem(TOKEN_KEY);
    this._token.set(null);
  }
}
```

### `TaskFlow.API/Controllers/ProjectsController.cs`
```csharp
using Microsoft.AspNetCore.Authorization;
using Microsoft.AspNetCore.Mvc;
using TaskFlow.BLL.DTOs;
using TaskFlow.BLL.Services.Interfaces;
using TaskFlow.DAL.Auth;

namespace TaskFlow.API.Controllers;

[ApiController]
[Route("api/[controller]")]
[Authorize]
public class ProjectsController(IProjectService projectService) : ControllerBase
{
    [HttpGet]
    public async Task<ActionResult<List<ProjectDto>>> Get(CancellationToken cancellationToken) =>
        Ok(await projectService.GetAllAsync(cancellationToken));

    [HttpGet("{id:int}")]
    public async Task<ActionResult<ProjectDto>> GetById(int id, CancellationToken cancellationToken)
    {
        var project = await projectService.GetByIdAsync(id, cancellationToken);
        return project is null ? NotFound() : Ok(project);
    }

    [HttpPost]
    [Authorize(Roles = AppRoles.Admin)]
    public async Task<ActionResult<ProjectDto>> Create([FromBody] ProjectUpsertDto dto, CancellationToken cancellationToken)
    {
        var created = await projectService.CreateAsync(dto, cancellationToken);
        return CreatedAtAction(nameof(GetById), new { id = created.Id }, created);
    }

    [HttpPut("{id:int}")]
    [Authorize(Roles = AppRoles.Admin)]
    public async Task<IActionResult> Update(int id, [FromBody] ProjectUpsertDto dto, CancellationToken cancellationToken) =>
        await projectService.UpdateAsync(id, dto, cancellationToken) ? NoContent() : NotFound();

    [HttpDelete("{id:int}")]
    [Authorize(Roles = AppRoles.Admin)]
    public async Task<IActionResult> Delete(int id, CancellationToken cancellationToken) =>
        await projectService.DeleteAsync(id, cancellationToken) ? NoContent() : NotFound();
}
```

## Verify it works

1. **Log in as a regular user** — the Projects link should not appear in the nav. Navigating directly to `/projects` should redirect to `/unauthorized`.
2. **Log in as an admin** — the Projects link should appear. Navigating to `/projects` should work.
3. **Use a tool like Postman or the browser DevTools** — send a `GET /api/projects` with a regular user's JWT. The API should return `403 Forbidden`.
4. **Send `GET /api/projects` with an admin JWT** → `200 OK` with the projects list.

> The backend `403` is the definitive proof that the security is enforced server-side, not just hidden client-side.

## Addendum — Bug fixes discovered during implementation

Three bugs were found and fixed as part of delivering this feature. They are documented here rather than in the original feature files to keep those files stable.

---

### Bug A — Login redirect defaulted to `/projects` for regular users

**File:** `src/app/pages/login/login.component.ts`

After login (or register), the component redirected the user to `returnUrl ?? '/projects'`. For a non-admin user this immediately hit the `adminGuard` and bounced them to `/unauthorized`, which looks like a broken login.

**Before:**
```ts
submit() {
  const returnUrl = this.route.snapshot.queryParamMap.get('returnUrl') ?? '/projects';
  // ...
  this.router.navigateByUrl(returnUrl);
}

register() {
  // ...
  this.router.navigateByUrl('/projects');
}
```

**After:**
```ts
submit() {
  const returnUrl = this.route.snapshot.queryParamMap.get('returnUrl') ?? '/';
  // ...
  this.router.navigateByUrl(returnUrl);
}

register() {
  // ...
  this.router.navigateByUrl('/');
}
```

Both fallbacks now go to `'/'` (the home page), which is accessible to all authenticated users. If the user was redirected to `/login` because they tried to visit a protected route, the `returnUrl` query parameter will still take them exactly where they wanted to go.

---

### Bug B — Soft-deleted records were returned by all repository queries

**File:** `TaskFlow.DAL/Repositories/Implementations/GenericRepository.cs`

The `BaseEntity` has an `IsDeleted: bool` field and a `SoftDelete()` method that sets it to `true`. However, the generic repository's `GetByIdAsync` and `GetAllAsync` methods did not filter on `IsDeleted`, so soft-deleted records were returned as if they were active.

**Before:**
```csharp
public Task<T?> GetByIdAsync(int id, CancellationToken ct = default)
    => DbSet.FirstOrDefaultAsync(x => x.Id == id, ct);

public Task<List<T>> GetAllAsync(CancellationToken ct = default)
    => DbSet.ToListAsync(ct);
```

**After:**
```csharp
public Task<T?> GetByIdAsync(int id, CancellationToken ct = default)
    => DbSet.FirstOrDefaultAsync(x => x.Id == id && !x.IsDeleted, ct);

public Task<List<T>> GetAllAsync(CancellationToken ct = default)
    => DbSet.Where(x => !x.IsDeleted).ToListAsync(ct);
```

The fix is a single predicate addition in each method. Because these are the only two read methods on the base repository, and all services go through the repository, this fix covers every entity in the system at once — projects, tasks, comments, tags.

---

### Bug C — Comment field name mismatch between frontend and backend

**File:** `src/app/core/api/task-items.api.ts`

The backend `TaskCommentDto` exposes the field as `text` (lowercase, following JSON camelCase conventions). The frontend `TaskItemDto.comments` array typed it as `content: string`, so every comment's text rendered as `undefined` in the UI.

**Before:**
```ts
export interface TaskItemDto {
  // ...
  comments: { id: number; content: string }[];
}
```

**After:**
```ts
export interface TaskItemDto {
  // ...
  comments: { id: number; taskItemId: number; text: string }[];
}
```

The field was renamed from `content` to `text` to match the API response. The `taskItemId` field was also added — it was present in the response but missing from the type, which would cause it to be silently dropped when spreading the object.
