# Step 3: Login Page Feature

This step covers the login/register page, including all TypeScript, HTML, and SCSS code.

## What & Why

This step builds the login and registration page. It lets users sign in or create an account, and handles form validation, error messages, and navigation. This is the entry point for users to access protected features in your app.

## 1. Login Component TypeScript (`src/app/pages/login/login.component.ts`)
```typescript
import { Component, inject, signal } from '@angular/core';
import { CommonModule } from '@angular/common';
import { FormBuilder, ReactiveFormsModule, Validators } from '@angular/forms';
import { Router, ActivatedRoute, RouterLink } from '@angular/router';
import { Observable } from 'rxjs';
import { AuthService } from '../../core/auth/auth.service';

@Component({
  selector: 'app-login',
  standalone: true,
  imports: [CommonModule, ReactiveFormsModule, RouterLink],
  templateUrl: './login.component.html',
  styleUrl: './login.component.scss'
})
export class LoginComponent {
  private readonly fb = inject(FormBuilder);

  readonly error = signal<string | null>(null);
  readonly busy = signal(false);

  readonly form = this.fb.nonNullable.group({
    email: ['', [Validators.required, Validators.email]],
    password: ['', [Validators.required, Validators.minLength(8)]]
  });

  constructor(
    private readonly auth: AuthService,
    private readonly router: Router,
    private readonly route: ActivatedRoute
  ) {}

  submit() {
    const returnUrl = this.route.snapshot.queryParamMap.get('returnUrl') ?? '/projects';
    this.authenticate(
      this.auth.login(this.form.getRawValue()),
      'Login failed. Check your email/password.',
      returnUrl
    );
  }

  register() {
    this.authenticate(
      this.auth.register(this.form.getRawValue()),
      'Register failed. The user may already exist.',
      '/projects'
    );
  }

  private authenticate(
    request$: Observable<unknown>,
    failureMessage: string,
    redirectTo: string
  ) {
    this.error.set(null);

    if (!this.ensureValidForm()) {
      return;
    }

    this.busy.set(true);
    request$.subscribe({
      next: async () => {
        await this.router.navigateByUrl(redirectTo);
      },
      error: () => {
        this.error.set(failureMessage);
        this.busy.set(false);
      },
      complete: () => this.busy.set(false)
    });
  }

  private ensureValidForm(): boolean {
    if (this.form.valid) {
      return true;
    }

    this.form.markAllAsTouched();
    return false;
  }
}
```

## 2. Login Component HTML (`src/app/pages/login/login.component.html`)
```html
<section class="card">
  <h1>Login</h1>
  <p class="muted">This calls <code>/api/auth/login</code> and stores the JWT in <code>localStorage</code>.</p>

  <form [formGroup]="form" (ngSubmit)="submit()">
    <label>
      <span>Email</span>
      <input type="email" formControlName="email" autocomplete="username" />
    </label>

    <label>
      <span>Password</span>
      <input type="password" formControlName="password" autocomplete="current-password" />
    </label>

    @if (error()) {
      <div class="error">{{ error() }}</div>
    }

    <div class="actions">
      <button type="submit" [disabled]="busy()">Sign in</button>
      <button type="button" class="secondary" (click)="register()" [disabled]="busy()">Register</button>
    </div>
  </form>
</section>
```

## 4. Adding the Login Route

Now that you have a Login component, you need to make it accessible via a URL. In Angular, this is done by adding a route to your `app.routes.ts` file:

```typescript
import { LoginComponent } from './pages/login/login.component';

export const routes: Routes = [
  // ...existing routes...
  { path: 'login', component: LoginComponent },
];
```

**What this does:**
- When users visit `/login`, Angular will display your Login page.
- You can now link to this route using `<a routerLink="/login">Login</a>` in your navigation or templates.

*We will **not** use the **Auth Guard** we've talked about earlier, because you don't need to be authenticated in order to login, right?*

**Tip:** If you already have a `routes` array, just add the new route object to it. If not, create one as shown above.

### Understanding the Angular Structure

This feature is split into three main files, each with a specific role in the Angular framework:

#### 1. TypeScript File (`login.component.ts`)
- **Purpose:** Contains the logic for the login/register page.
- **Structure:**
  - **Imports:** Brings in Angular modules, services, and utilities needed for forms, routing, and authentication.
  - **Component Decorator:** Defines metadata like selector, template, and styles.
  - **Class:** Holds the logic:
    - **Form Setup:** Uses Angular’s `FormBuilder` to create a reactive form with validation.
    - **Signals:** Manages UI state (like errors and loading) reactively.
    - **Methods:** Handles login, registration, form validation, and navigation.
- **Why:** Separates business logic from the UI, making the code easier to test and maintain.

#### 2. HTML File (`login.component.html`)
- **Purpose:** Defines the structure and content of the login/register page.
- **Structure:**
  - **Form Elements:** Inputs for email and password, submit and register buttons.
  - **Error Display:** Shows validation or server errors to the user.
  - **Bindings:** Connects form controls and button states to the TypeScript logic.
- **Why:** Keeps the UI markup clean and separate from logic, following Angular’s component-based approach.

#### 3. SCSS File (`login.component.scss`)
- **Purpose:** Styles the login/register page for a modern, user-friendly look.
- **Structure:**
  - **Card Layout:** Centers the form and gives it a distinct background.
  - **Form Styling:** Spacing, colors, and button styles for clarity and accessibility.
  - **Error Styling:** Highlights errors so users can easily spot issues.
- **Why:** SCSS allows for modular, reusable, and maintainable styles scoped to this component.

**Summary:**
- This separation of concerns (logic, template, style) is a core Angular pattern. It helps you build scalable, maintainable apps where each part of a feature is easy to find and update.

---

### How the Three Files Work Together

- The **TypeScript file** defines the component class, which holds all the logic and data for the page. It exposes properties (like `form`, `error`, `busy`) and methods (like `submit()` and `register()`) that the template can use.
- The **HTML file** (template) binds to these properties and methods using Angular’s template syntax. For example, `(ngSubmit)="submit()"` calls the `submit()` method when the form is submitted, and `[formGroup]="form"` binds the form controls to the TypeScript logic.
- The **SCSS file** styles the HTML elements, but does not interact with the TypeScript directly. It only affects how things look.

**In summary:**
- The HTML template “talks” to the TypeScript class by binding to its properties and calling its methods. This is how user actions in the UI trigger logic in your code.
- The SCSS file makes everything look good, but doesn’t affect the logic or data.
